# 视频中 PPT 图片内容识别需求

本文描述“视频中展示的 PPT 页面里，图片、图表、示意图等非文字内容的识别与索引”需求。它不是从 `.pptx` 文件中提取原始图片，也不要求把视频画面中的图片区域裁剪出来；核心目标是让这些视觉内容变得可搜索、可总结、可被后续问答使用。

## 背景

当前系统已经支持从视频中提取字幕、音频转写，以及可选的视频画面 OCR。OCR 能覆盖 PPT 页面上的显式文字，但无法理解页面中的图片内容，例如：

- 架构图、流程图、模块关系图
- 折线图、柱状图、饼图等数据图表
- 产品截图、网页截图、代码界面截图
- 照片、插图、示意图、实验图
- 只包含少量文字但信息主要来自图像关系的页面

这些内容往往是课程、发布会、技术分享和产品演示视频的关键知识点。如果只索引字幕和 OCR 文字，搜索“API 网关架构”“GPU 集群拓扑”“销售趋势下降原因”等问题时，系统可能无法命中相关页面。

## 目标

新增一种可选的视觉理解能力，对视频中的稳定 PPT 页面生成结构化描述，重点识别页面里的图片、图表和示意关系。

目标能力：

- 复用当前视频抽帧、稳定页面检测、页面去重和 OCR 结果。
- 对每个去重后的代表 PPT 页面生成 `visual_summary`。
- 生成适合搜索召回的 `visual_keywords`。
- 对图表、流程图、架构图等内容尽量输出结构化解释。
- 输出结果与现有 `pages.json` 合并，方便索引和展示。
- 控制在线 API 调用量，禁止逐帧调用视觉模型。

期望效果示例：

```json
{
  "page_index": 12,
  "start": 318.0,
  "end": 342.0,
  "text": "OCR 识别出的页面文字",
  "visual_summary": "页面中部是一张系统架构图，展示客户端请求经过 API 网关进入模型服务，再访问缓存和数据库。箭头表示请求链路，底部监控模块覆盖所有服务。",
  "visual_keywords": ["系统架构", "API 网关", "模型服务", "缓存", "数据库", "监控"],
  "visual_content_type": "architecture_diagram",
  "visual_confidence": 0.86,
  "frame_path": "frames/page_0012.jpg"
}
```

## 非目标

第一阶段不解决以下问题：

- 不从 `.pptx` 源文件中提取原始图片。
- 不保证把 PPT 中的图片区域单独裁剪出来。
- 不恢复图片原始分辨率、透明背景、原始文件名或图层结构。
- 不做完整 PPT 重建。
- 不对视频的每一帧都做视觉模型分析。
- 不保证复杂图表中的所有数字都能被精确识别，精确数值仍以 OCR 或人工校验为准。

如果后续确实需要图片裁剪，可以在代表页截图的基础上增加区域检测，但这不是本次需求的前置条件。

## 触发方式

建议新增独立开关：

```python
extract_visual: bool = False
describe_visual: bool = False
```

语义：

- `extract_visual=True`：提取视频画面文字，生成 `pages.json`。
- `describe_visual=True`：在代表页基础上追加视觉内容理解。
- `describe_visual=True` 应隐式要求 `extract_visual=True`，因为它依赖代表页检测和页面去重结果。

前端可以呈现为两个选项：

- 提取视频画面文字
- 识别 PPT 图片和图表内容

第二个选项开启时，应自动开启第一个选项或提示依赖关系。

## 推荐流程

整体流程不要改变当前的字幕和转写主链路，只在视觉提取之后增加一步描述。

```text
视频文件
 ↓
抽帧
 ↓
检测文字/PPT 页面
 ↓
页面变化检测和稳定帧选择
 ↓
OCR
 ↓
页面文本去重，生成代表页
 ↓
如果 describe_visual=True：
  - 对代表页截图做质量检查
  - 按 hash 查缓存
  - 缓存未命中时调用视觉模型
  - 写入 visual_summary / visual_keywords / visual_content_type
 ↓
写入 pages.json 和 meta.json
```

关键约束：

- 只处理去重后的代表页，不处理原始 frames。
- 代表页数量过多时需要有上限，例如 `max_visual_describe_pages=120`。
- 同一帧或高度相似帧必须通过 hash 缓存复用结果。
- 视觉描述失败时可以降级为仅 OCR 成功，具体是否失败任务应由配置控制。

## 本地模型与在线 API

### 本地模型

本地视觉语言模型可以降低外部依赖和隐私风险，适合做粗粒度识别：

- 页面是否包含图表、照片、流程图、截图
- 生成简单图片描述
- 对低敏感度任务做离线处理

主要不足：

- 对低清晰度视频帧、小字号图表和复杂界面截图不稳定。
- 结构化 JSON 输出一致性较弱。
- 对图表趋势、坐标轴、架构关系的解释容易漏项或幻觉。
- 部署成本较高，需要 GPU、模型管理和推理服务。

### 在线 API

在线多模态 API 更适合生成高质量可搜索描述，尤其是图表、流程图、产品截图和复杂页面。成本风险主要来自调用数量，而不是单页成本。

本需求的成本控制原则：

- 不把完整视频或所有 frames 发送给 API。
- 本地完成抽帧、PPT 页检测、OCR、去重和缓存。
- API 只接收最终代表页截图。
- 优先使用低成本视觉模型。
- 对低置信度或复杂图表再选择性升级到更强模型。
- 支持批处理模式，允许牺牲实时性换取更低成本。

## 成本控制策略

必须实现以下策略，避免视频较长时成本失控：

1. **代表页调用**
   - 原始视频即使有几千帧，也只对去重后的 PPT 页面调用视觉模型。
   - 1 小时课程视频通常应压缩到几十到一两百个代表页。

2. **图像质量过滤**
   - 过滤黑屏、过渡帧、模糊帧、讲师大面积遮挡帧。
   - 对低质量帧优先尝试在相邻稳定区间选择更清晰的代表帧。

3. **相似帧缓存**
   - 对代表帧计算感知 hash。
   - hash 相同或相似度高时复用已有视觉描述。
   - 缓存 key 应包含模型、prompt 版本和图片 hash。

4. **每任务上限**
   - 增加 `max_visual_describe_pages`。
   - 超过上限时按页面质量、OCR 信息量、时间分布采样。
   - 在 `meta.json` 中记录被跳过的页数和原因。

5. **短输出**
   - 提示模型输出短 JSON。
   - 控制 `visual_summary` 长度。
   - 关键词数量建议 5 到 12 个。

6. **分级模型**
   - 默认使用便宜模型。
   - 只有当页面疑似复杂图表、架构图或低置信度时，再用更强模型重试。

## 输出字段

建议扩展 `pages.json.pages[]`：

```json
{
  "page_index": 0,
  "start": 12.0,
  "end": 36.0,
  "text": "OCR text",
  "frame_path": "frames/page_0000.jpg",
  "confidence": 0.91,
  "visual_summary": "页面右侧是一张折线图，展示 2021 到 2024 年收入持续增长，2024 年增速最高。",
  "visual_keywords": ["收入趋势", "折线图", "增长", "2021-2024"],
  "visual_content_type": "chart",
  "visual_confidence": 0.82,
  "visual_provider": "configured-provider",
  "visual_model": "configured-model",
  "visual_cache_hit": false
}
```

`visual_content_type` 建议枚举：

- `none`
- `photo`
- `chart`
- `table`
- `flowchart`
- `architecture_diagram`
- `screenshot`
- `illustration`
- `mixed`
- `unknown`

`meta.json` 建议增加：

```json
{
  "visual_description": {
    "enabled": true,
    "provider": "configured-provider",
    "model": "configured-model",
    "candidate_pages": 86,
    "described_pages": 72,
    "cache_hits": 14,
    "skipped_pages": 0,
    "failed_pages": 0
  }
}
```

## Prompt 要求

视觉模型提示词应强调：

- 重点描述图片、图表、流程图、架构图、截图和示意关系。
- 不要重复 OCR 已经识别出的普通页面标题和正文。
- 如果图表可理解，描述趋势、对比关系和关键结论。
- 如果是架构图，描述组件、连接关系、数据流或调用链路。
- 如果是产品截图，描述界面类型、关键模块和用户操作含义。
- 不确定时明确降低置信度，不编造看不清的细节。
- 输出严格 JSON，便于写入 `pages.json`。

输入应包含：

- 代表帧图片
- OCR 文本
- 页时间范围
- 输出 schema

## 错误处理

建议默认策略：

- OCR 和代表页生成失败：沿用现有视觉提取失败策略。
- 视觉描述 API 单页失败：记录该页失败，继续处理其他页。
- API 全局不可用：任务可配置为失败或降级。
- 输出 JSON 解析失败：重试一次；仍失败则记录原始错误并跳过该页。

建议新增配置：

```python
describe_visual: bool = False
visual_description_optional: bool = True
max_visual_describe_pages: int = 120
visual_description_provider: str | None = None
visual_description_model: str | None = None
```

`visual_description_optional=True` 时，视觉描述失败不应影响字幕、转写和 OCR 结果产出。

## 隐私和合规

如果使用在线 API，需要在产品和配置层明确：

- 视频截图可能被发送到第三方模型服务。
- 用户应能关闭该功能。
- 敏感内容场景建议只使用本地模型或禁用视觉描述。
- 日志中不要写入完整图片内容或 API 原始响应。
- 缓存图片 hash 和结构化结果即可，不应额外复制敏感图片到非输出目录。

## 分阶段实现

建议分三期：

### 第一期：代表页视觉描述

- 新增 `describe_visual` 配置链路。
- 复用现有 `pages.json` 代表帧。
- 实现视觉描述器接口和一个在线 API provider。
- 增加缓存、页数上限和失败降级。
- 将 `visual_summary`、`visual_keywords` 写入 `pages.json`。

### 第二期：本地筛选增强

- 增加图片质量评分。
- 增加页面内容类型初筛。
- 对明显纯文字页跳过 API。
- 对复杂图表页标记为可升级模型处理。

### 第三期：高级图像区域理解

- 如有必要，再检测页面中的图像区域。
- 对单独图片区域或图表区域做二次描述。
- 支持更细粒度的图片索引和页面内定位。

## 验收标准

功能验收：

- 开启 `describe_visual` 后，代表页中至少包含 `visual_summary` 和 `visual_keywords` 字段。
- 不对原始 frames 逐帧调用视觉模型。
- 相同代表帧重复运行时命中缓存。
- 超过页数上限时任务仍成功，并在 `meta.json` 记录跳过信息。
- API 单页失败不会导致整条任务失败，除非配置为强制失败。

质量验收：

- 架构图页面能描述主要模块和连接关系。
- 流程图页面能描述流程顺序。
- 图表页面能描述明显趋势或对比关系。
- 产品截图页面能描述界面和主要功能区域。
- 纯文字页面不应生成冗余的长视觉描述。

成本验收：

- 1 小时视频不得按全部抽帧数量调用在线 API。
- API 调用数量应接近去重后的 PPT 页面数。
- 默认每任务有最大视觉描述页数限制。
- 缓存命中率和实际调用数应写入任务元数据。
